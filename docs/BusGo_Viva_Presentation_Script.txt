================================================================================
  SECTION 2 — PROBLEM STATEMENT & PROJECT CONTEXT
================================================================================
  [Duration: ~2 minutes]
--------------------------------------------------------------------------------

Sri Lanka's public bus network is one of the most heavily used transport
systems in the country, yet it operates almost entirely without digital support.

  • Commuters have no way to know where a bus is, how full it is, or when
    it will arrive.
  • Drivers have no digital tools to report emergencies or log trips.
  • Administrators manage fleets manually with no real-time visibility.

BusGo solves all three problems simultaneously.

The system provides:

  • Real-time bus tracking on live maps for both passengers and administrators.
  • A structured digital workflow for drivers to log trips, report emergencies,
    and receive route information.
  • A QR-code-based ticketing system for quick and paperless boarding.
  • A centralised admin control panel to oversee the full fleet, manage users,
    respond to emergencies, and maintain complete audit trails.

The system is built around three Colombo metropolitan bus routes:
  • Route 138 — Nugegoda to Colombo Fort
  • Route 163 — Rajagiriya to Maharagama
  • Route 171 — Colombo 4 to Athurugiriya

These routes were chosen because of their high ridership and geographic
coverage across the Colombo district.

  4.1  TECHNOLOGY STACK — ADMIN PANEL
────────────────────────────────────────────────────────────────────────────────

For the Admin Panel, I chose the following stack:

  React 19 + TypeScript
    React is the industry standard for building component-based web UIs.
    TypeScript was chosen to catch type errors at compile time rather than
    at runtime — critical where API data shapes are complex.

  Vite
    Used as the build tool for extremely fast hot module replacement during

  React Router v7
    For client-side routing with protected routes that redirect
    unauthenticated users.

  React Leaflet
    For rendering interactive maps. Leaflet was chosen over Google Maps
    because it is open-source, requires no billing-enabled API key, and
    integrates with OpenStreetMap tiles.

  Axios
    For API communication with the backend, wrapped in a central typed
    service layer.

  Lucide React
    A consistent, lightweight icon library used throughout the UI.

────────────────────────────────────────────────────────────────────────────────
  4.2  APPLICATION STRUCTURE
────────────────────────────────────────────────────────────────────────────────

The admin panel is structured around a protected route system. Any user who
is not authenticated is automatically redirected to the login page. Once
logged in, the panel uses a persistent sidebar layout with a main content
area — giving the feel of a native desktop application.

The eight routes are:

  /admin/login        Authentication page
  /admin/dashboard    The main control centre
  /admin/fleet-map    Full-screen live map
  /admin/emergencies  Emergency alert management
  /admin/fleet        Bus fleet management
  /admin/users        User management (drivers, passengers, admins)
  /admin/routes       Route and bus stop management
  /admin/audit-logs   Admin action audit trail

Authentication is implemented using React Context API. The AuthContext stores
the admin session, handles login and logout, and persists the access token in
localStorage. Every protected route checks the context before rendering —
unauthenticated access always redirects to login.

────────────────────────────────────────────────────────────────────────────────
  4.3  DASHBOARD PAGE
────────────────────────────────────────────────────────────────────────────────

The Dashboard is the first screen an admin sees after logging in. It is
designed to give an immediate, at-a-glance overview of the entire bus network.

Key features:

    Show active buses, passengers today, pending emergency alerts, and standby

    I used polling with setInterval rather than WebSockets for statistics.

  SECTION 6 — KEY TECHNICAL DECISIONS & JUSTIFICATIONS
================================================================================
  [Duration: ~3 minutes | Be ready to defend every choice listed here]
--------------------------------------------------------------------------------

  1. Why React for the Admin Panel and Flutter for mobile?

     React is the dominant library for complex, data-driven web applications.
     Its component model and ecosystem made it ideal for a multi-page dashboard
     with interactive tables, modals, and maps. Flutter was chosen for mobile
     because it compiles to native ARM code, giving smooth 60fps animations
     critical for map-heavy apps, and one codebase targets both Android and iOS.

  2. Why Provider over Bloc or Riverpod in Flutter?

     Provider is lightweight, integrates naturally with Flutter's widget tree
     via InheritedWidget, and is officially recommended by the Flutter team.
     Our app has five distinct state domains (Auth, Bus, Trip, User, Emergency)
     and Provider's multi-provider pattern kept each isolated and testable.
     Bloc or Riverpod would have added boilerplate without benefit at our scale.

  3. Why React Context API for authentication instead of Redux or Zustand?

     Authentication state is simple — authenticated or not — and changes
     rarely. Context API with a single AuthProvider is the right fit. Redux
     would have been over-engineering for this specific use case.

  4. Why OpenStreetMap / Leaflet / flutter_map instead of Google Maps?

     Google Maps requires a billing-enabled API key, introducing cost
     uncertainty. OpenStreetMap with Leaflet (web) and flutter_map (mobile) is
     fully open-source and provides the same tile quality for our Colombo use
     case. OSRM provides road-accurate polylines at no cost.

  5. Why Supabase Realtime instead of a custom WebSocket server?

     Supabase Realtime is a managed, scalable pub/sub infrastructure we connect
     to directly from Flutter without building and maintaining a custom
     WebSocket server. For bus location broadcasting — where the backend pushes
     updates to all subscribed clients — this is exactly what Realtime is
     designed for.

  6. Why TypeScript over plain JavaScript for the admin panel?

     The admin panel consumes multiple API endpoints with complex data shapes:
     buses, drivers, passengers, emergency alerts, routes, stops, audit logs.
     TypeScript enforces correct handling of each shape at compile time,
     catching mismatches before they reach production. For a production-
     readiness goal, TypeScript was non-negotiable.




================================================================================
  SECTION 7 — CHALLENGES & HOW I SOLVED THEM
================================================================================
  [Duration: ~2–3 minutes | This section shows your problem-solving ability]
--------------------------------------------------------------------------------

  Challenge 1: Animating buses along road-accurate polylines in Flutter
  ─────────────────────────────────────────────────────────────────────
  The live map needed buses moving smoothly along actual road paths, not
  straight lines. The OSRM API returned polylines as lists of lat/lng
  coordinates, but animating a marker along hundreds of waypoints required a
  custom algorithm.

  Solution:
    I computed the cumulative distance of the entire polyline, then advanced
    a progress value (0.0 to 1.0) each timer tick. The current position is
    found by identifying which segment the progress falls into and linearly
    interpolating between those two coordinates. This gives smooth, continuous
    movement that follows the actual road shape.

  Challenge 2: Displaying multiple route polylines on the admin panel map
  ────────────────────────────────────────────────────────────────────────
  The admin panel needed multiple route polylines simultaneously, each with
  a different colour and each linked to its own stops and buses. Loading all
  routes, stops, and road polylines involves multiple async API calls, and a
  naive approach would produce partial or flickering renders.

  Solution:
    I designed a RouteLayer data structure that bundles the route ID, colour,
    stop list, and polyline coordinates together. All routes are loaded in
    parallel using Promise.all. Once all data is ready, the map renders all
    layers atomically — no flickering, no partial states.

  Challenge 3: Keeping the admin panel fresh without a global state manager
  ──────────────────────────────────────────────────────────────────────────
  With eight pages each making their own API calls, I needed data to be fresh
  without over-fetching or duplicating logic.

  Solution:
    Each page fetches its own data on mount via useEffect. Polling with
    setInterval is used only where live data is genuinely needed (Dashboard
    and Emergencies). All service calls are centralised in a typed service
    layer so no logic is ever duplicated across pages.

  Challenge 4: Making the scanner viewfinder feel professional
  ────────────────────────────────────────────────────────────
  A scanner screen that feels like a real product needs precise visual
  feedback — it cannot look like a plain camera view.

  Solution:
    I implemented two independent AnimationControllers:
      • Scan line  — 2400ms duration, easeInOut, repeating
      • Pulse glow — 1500ms duration, linear, repeating
    Layering these with Stack and Positioned widgets produced the professional
    scanning UI without any third-party animation library.

  ┌─────────────────────────────────────────────────────────────────────────┐
  │ Q1: Why did you use React for the admin panel instead of a mobile app?  │
  └─────────────────────────────────────────────────────────────────────────┘

  The admin panel is designed for desktop use by transport administrators who
  sit at workstations. A web application gives them a large screen with
  multiple data panels visible simultaneously — something a mobile screen
  cannot offer. React's component-based architecture also made it
  straightforward to build complex multi-panel layouts like the split-view
  Fleet Map page.

  ┌───────────────────────────────────────────────────────────────────────────┐
  │ Q2: What is the difference between the three Flutter apps?                │
  └───────────────────────────────────────────────────────────────────────────┘

  They serve three completely different user roles.

  The Passenger app (BusGo Client) is for the general public — finding buses,
  tracking them live, and holding a digital QR ticket.

  The Driver app (BusGo Drive) is for the person driving the bus — showing
  route information, passenger counts, and enabling SOS reporting during a trip.

  The Scanner app (BusGo Scanner) is for the conductor — its sole purpose is
  scanning passenger QR codes to validate tickets.

  Each app is scoped tightly to its user's role to keep the UI focused and
  easy to use in their specific context.

  ┌──────────────────────────────────────────────────────────────────────────┐
  │ Q3: How does the QR ticketing system work end to end?                    │
  └──────────────────────────────────────────────────────────────────────────┘

  When a passenger registers, the backend generates a unique QR code string
  for their account and stores it in their profile. The Passenger app's QR
  Card screen fetches this string via the UserProvider and renders it as a QR
  code image using the qr_flutter package. When the conductor opens the Scanner
  app and points it at the passenger's screen, the app reads the QR value and
  verifies it against the backend. A valid code shows the Scan Success screen;
  an invalid or expired code shows the Scan Error screen.

  ┌──────────────────────────────────────────────────────────────────────────┐
  │ Q4: How does the live map work? Is it actually real-time?                │
  └──────────────────────────────────────────────────────────────────────────┘

  The system has two layers of real-time data.

  In the mobile apps, Supabase Realtime subscribes to bus location broadcasts
  pushed by the backend. Additionally, I implemented a client-side animation
  layer: even between Realtime updates, buses appear to move smoothly because
  the app animates markers along the known route polyline using timer-based
  interpolation.

  On the admin panel, bus positions are fetched through polling every few
  seconds via the REST API.

  For a production deployment with GPS hardware installed in buses, the
  Supabase Realtime channel would receive live GPS coordinates and broadcast
  them instantly to all subscribers.

  ┌──────────────────────────────────────────────────────────────────────────┐
  │ Q5: How did you handle authentication and security on the front end?     │
  └──────────────────────────────────────────────────────────────────────────┘

  In the admin panel, authentication uses JWT tokens stored in localStorage.
  The AuthContext attaches the access token to every Axios request header via
  a central Axios instance. If a request returns 401 Unauthorised, the user
  is redirected to the login page immediately.

  In the Flutter apps, tokens are stored using flutter_secure_storage, which
  uses the device's secure enclave — Android Keystore on Android and iOS
  Keychain on iOS. Tokens cannot be read by other apps or through file system
  access. The TokenService wraps all token read/write operations, and the
  ApiClient automatically injects the token into every Dio request header.

  ┌──────────────────────────────────────────────────────────────────────────┐
  │ Q6: What would you improve if you had more time?                         │
  └──────────────────────────────────────────────────────────────────────────┘

  Several things.

  First, I would integrate actual GPS hardware tracking through the driver app
  so real bus positions replace the simulated movement.

  Second, I would add push notifications to the passenger app using Firebase
  Cloud Messaging so passengers receive alerts when their bus is approaching.

  Third, I would implement offline support in the Flutter apps using a local
  SQLite cache so the app remains usable in low-connectivity areas — a real
  concern in suburban Sri Lanka.

  Fourth, I would add end-to-end tests for critical user flows: login, QR
  scanning, and emergency submission.

  ┌──────────────────────────────────────────────────────────────────────────┐
  │ Q7: How did you collaborate with your team as the front-end developer?   │
  └──────────────────────────────────────────────────────────────────────────┘

  My work depended entirely on REST API contracts provided by the backend team.
  We agreed on API request and response formats early in the project, and I
  formalised these contracts in a typed service layer — TypeScript interfaces
  in React and Dart model classes in Flutter.

  This meant I could develop front-end features against consistent interfaces
  even before backend endpoints were fully implemented, by using mock data
  services. Once the backend was ready, I swapped mock services for real API
  calls with minimal changes to the UI layer.
